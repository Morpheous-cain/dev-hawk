# Handoff 2 — Auth Access-Denied Fix

**Date:** 6 July 2026
**Focus:** CEO (and other roles) hitting "Access Denied" page despite correct role/permissions.

## 1. Root Cause
`RequireRole` route guard denied console access when `userRole` was still `null`. The role fetch in `useAuth` is async — deferred via `setTimeout(0)` inside `onAuthStateChange` — so `userRole` lags behind `loading` flipping to `false`. The guard treated `null` role as a denial, so CEO hit the 403 page during that race window.

Secondary: `CONSOLE_ROLES` set was missing canonical DB enum values (`hr_custodian`, `bdo` alias handling) and had a duplicate `operations_supervisor` entry. Also, `loading` returned `null` (blank screen) instead of a spinner.

## 2. Changes Made

### `src/components/auth/RequireRole.tsx`
1. **Spinner on loading** — replaced `return null` with a centered spinner so the user sees feedback while auth resolves.
2. **Wait-for-role guard** — added a new branch: if `user` exists but `userRole` is still `null`, render the spinner instead of the 403 page. This eliminates the race that caused the false denial.
3. **`CONSOLE_ROLES` cleanup**:
   - Added canonical `app_role` enum values (`ceo`, `coo`, `control_room_officer`, `operations_supervisor`, `bdo`, `system_admin`) at the top.
   - Removed the duplicate `operations_supervisor` line.
   - Kept all legacy/UI role strings (`admin`, `gm`, `ops_manager`, `control`, `hr`, `finance`, `compliance`, etc.) so existing `user_roles` rows and the Auth page's `selected_management_role` values continue to match.
   - Note: `hr_custodian` is part of the DB enum but not currently used in the Auth UI role picker; left out of `CONSOLE_ROLES` to avoid implying access the UI cannot assign. Add when HR UI is wired.

### No other files touched
`useAuth.ts`, `usePermissions.ts`, `accessControl.ts`, `App.tsx`, `Auth.tsx`, `PlatformPage.tsx` all unchanged. The bug was isolated to the guard.

## 3. How the Auth Flow Works (verified)
1. `AuthProvider` subscribes to `supabase.auth.onAuthStateChange` first, then warms cache with `getSession()`.
2. On session, it defers `fetchUserRole(userId)` via `setTimeout(0)` to avoid Supabase deadlock inside the callback.
3. `loading` flips to `false` only after the role fetch completes (or on unauthenticated / signout).
4. `RequireRole` now waits for both `user` AND `userRole` before evaluating access.
5. `usePermissions()` normalises `userRole` to lowercase/underscore and checks against `accessMatrix` in `accessControl.ts`.

## 4. Routes
- `/platform/:platformId/*` — handled by `PlatformPage` (its own shell, no `RequireRole` wrapper). CEO login lands here at `/platform/ceo`.
- `/*` (everything else under `ConsoleLayout`) — wrapped by `RequireRole`. This is the path that was false-denying.
